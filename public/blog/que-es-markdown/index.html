¿Qué es Markdown? Guía práctica

Markdown es una forma sencilla de escribir texto con formato usando solo caracteres. Si has visto un README en GitHub, probablemente ya lo usaste. En esta guía te explico lo esencial, lo que la gente suele confundir y cómo usarlo en la vida real (docs, notas, blogs, producto y desarrollo).

TL;DR

  • Markdown es texto plano con estructura: headings, listas, links y código.
  • Es portable, rápido y perfecto para docs y trabajo en equipo.
  • Si dominas headings + listas + bloques de código, ya estás en el 80%.

1. ¿Qué es Markdown (y qué no)?

Markdown es un lenguaje de marcado ligero. Su objetivo no es "diseñar" sino estructurar: títulos, listas, links, código, citas. Luego una herramienta (GitHub, un editor, un generador de sitio) lo convierte a HTML u otro formato.

No es un reemplazo de Figma, ni un sistema de estilos. Tampoco es un estándar único: existen variantes (CommonMark, GitHub Flavored Markdown, Markdown Extra). A nivel práctico: si escribes para web, GitHub o notas, casi siempre te basta con la sintaxis "común" + dos o tres extras.

2. Por qué sigue siendo tan útil

  • Portable: es texto plano. Vive bien en git, en un ticket, en una wiki o en un repo.
  • Rápido: escribes sin pelearte con toolbars o estilos.
  • Legible: incluso sin renderizar, se entiende.
  • Automatizable: se transforma a HTML/PDF, se valida, se versiona, se revisa en PRs.
  • IA-friendly: es una estructura clara para prompts, especificaciones y documentación.

3. Sintaxis esencial con ejemplos

Títulos

# Titulo principal
## Seccion
### Sub-seccion

Énfasis

**negrita**
*cursiva*
`codigo inline`

Listas

- Item
- Item
  - Subitem

1. Paso uno
2. Paso dos

Links e imágenes

[Texto del link](https://example.com)

![Alt de la imagen](/ruta/imagen.png)

Citas

> Una cita corta.
>
> Puede tener varias lineas.

Bloques de código

Los bloques de código son clave para documentar APIs, comandos, pasos y ejemplos. Muchos renderizadores soportan un "lenguaje" para resaltar sintaxis. 

```bash
npm run build
```

```ruby
class User < ApplicationRecord
end
```

4. Casos de uso reales (no teóricos)

README de un proyecto

  • Qué hace el proyecto
  • Requisitos
  • Instalación / Setup
  • Comandos frecuentes
  • Decisiones técnicas

Especificaciones rápidas

Markdown es excelente para escribir una spec que puedas revisar en PR: alcance, casos, edge cases, criterios de aceptación, riesgos. Si estás definiendo el alcance de un producto, esto se complementa bien con un enfoque de MVP con criterio.

Notas de arquitectura

Para ADRs o notas técnicas, Markdown te da estructura sin fricción. Lo importante es la claridad, no el formato.

5. Errores comunes

  • Usar Markdown como si fuera diseño: intentar controlar pixeles, fuentes o layout fino.
  • Estructura pobre: no usar headings, hacer todo en párrafos largos.
  • Links sin contexto: "haz click aquí" en lugar de texto descriptivo.
  • Mezclar HTML innecesario: solo cuando el renderizador lo requiera.

6. Flujo práctico recomendado

  1. Escribe el contenido primero (títulos + bullets).
  2. Ordena: de lo general a lo específico.
  3. Agrega ejemplos cortos y copiables.
  4. Pasa un corrector ortográfico, y luego revisa consistencia de headings.

7. Herramientas que valen la pena

  • GitHub: issues/PRs/README.
  • VS Code: extensiones + preview.
  • Obsidian: notas y conocimiento personal.
  • Notion: útil, pero ojo con export/portabilidad.

Preguntas frecuentes

¿Markdown es un lenguaje de programación?

No. Es un formato de texto para estructurar contenido. No ejecuta lógica.

¿Me sirve si no soy dev?

Sí. Es ideal para notas, documentación, briefs y contenido que quieras versionar o compartir.

¿Qué variante debo usar?

Si escribes para GitHub, usa GitHub Flavored Markdown. Para el resto, CommonMark suele ser suficiente.

Cierre

Si solo aprendes tres cosas hoy: usa headings, usa listas, y usa bloques de código. Eso solo ya hace tu documentación más clara, revisable y mantenible.
